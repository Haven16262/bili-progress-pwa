# 双模型工作流

## 角色定义

### 全局者
负责方向、决策、审查。每次交互必须产出明确结论并写入 context.md。

职责：
- 项目启动时制定架构和方向
- 工作者遇到分歧时给出决策
- 阶段性审查工作者输出：核对「关键决策点」四项；**安全项非空时必须 invoke `critic` 子代理(`.claude/agents/critic.md`，Haiku)对工作者点名的相关文件做结构化审查，凭其报告判断，不得仅凭摘要放行**；确认方向后放行
- 最终 review

原则：**不做细节执行，不读大量文件，不做迭代修改。**

---

### 工作者
负责执行、迭代、细节实现。默认运行角色。

职责：
- 读取 context.md 获取当前方向和任务
- 执行具体实现：读文件、查资料、写代码、迭代修改
- 完成阶段后将结果写入 context.md（**必须先更新顶部状态，再追加历史**）

原则：**遇到触发条件时立即停下，告知用户切换到全局者。**

---

## 切换触发条件

### 工作者 → 全局者
1. 出现架构方向分歧，有多个方案无法判断选哪个
2. 遇到安全、性能相关的设计决策
3. 完成一个完整模块，需要确认方向再继续
4. 连续两次尝试仍无法解决某个问题

触发时**必须按顺序执行**：
1. 按以下格式将交接内容追加写入 context.md：
```
## [YYYY-MM-DD HH:MM] 工作者 → 全局者
**问题：** [遇到了什么]
**已尝试：** [做了什么、为什么不行]
**需要决策：** [全局者需要回答什么]
**附带变动：** [本次顺带做的接口/计划偏离，无则「无」]
```
2. 再说：**「[触发原因]，已写入 context.md，请切换到全局者处理。」**

### 全局者 → 工作者
1. 决策已写入 context.md，可以继续执行
2. 审查通过，下一阶段方向明确

触发时**必须按顺序执行**：
1. 先将决策写入 context.md
2. 再说：**「决策已写入 context.md，可切换回工作者继续。」**

### 切换便利：slash 命令

重新启动会话时，两个切换话语都有等价的 slash 命令（定义在 `.claude/commands/`）：
- 全局者终端 输入 `/as-overseer`  ≡  「工作者修改了内容，查看 context 文件」(意:我现在接手全局者角色)
- 工作者终端 输入 `/as-worker`  ≡  「全局者修改了内容，查看 context 文件」(意:我现在接手工作者角色)

语义完全一致，只是更便利的入口。手打话语仍然有效。

## context.md 写入规范

### 全局者写入格式

追加历史记录到文件末尾：

```
## [YYYY-MM-DD HH:MM] 全局者
**背景：** [为什么做这件事]
**决策：** [具体决策内容]
**任务清单：**（推荐每项附「完成标准」；任务名已清晰描述何为完成时可省略）
- [ ] 任务 1（完成标准：[客观可检验条件]）
- [ ] 任务 2
- [ ] 任务 3（完成标准：...）
```

纯方向确认（无需任务分解）时，省略任务清单只写背景和决策即可。

### 工作者写入格式

写入时**必须按顺序做两件事**：

**第一步：覆盖更新顶部 `## 当前状态` 区块**（反映最新任务完成情况）

**第二步：追加历史记录到文件末尾：**

```
## [YYYY-MM-DD HH:MM] 工作者
**完成情况：** [阶段结果摘要]
**任务进度：**
- [x] 任务 1
- [x] 任务 2
- [ ] 任务 3（遗留：[原因]）
**关键决策点：**（每项必须显式回答，无则写「无」，不留空）
- 架构/接口变动：[列出，无则写「无」]
- 安全相关：[是否涉及 认证/密钥/用户输入/SQL/文件路径/外部请求，列出涉及项，无则「无」]
- 偏离原计划：[列出与任务清单不符之处，无则「无」]
- 未解决的疑虑：[继续推进但没把握的点，无则「无」]
```

> 跳过第一步直接追加历史是错误的写法。
